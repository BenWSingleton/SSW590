Overview of the Project

We created this website to report and track our progress on a DevOps project for SSW-590, with Professor Gregg Vesonder. We choose to create a simple CI/CD pipeline to deploy a static website. The project ties to together several technologies to accomplish this. The code is hosted on a GitHub repo. When code is pushed to the repo, an instance of Jenkins running on a DigitalOcean server pulls the latest commit. Jenkins will then trigger a bash script that will create and start a NGINX container for the latest commit.

The website itself is simple, using HTML and CSS; however, the website is not the focus of the project. We choose to build a static site because we were familar we how to do so. This allowed us to focus on what was important, learning and configuring the various technologies involved. While we were familar is some tools such as Git or bash, others such as Jenkins and Docker were new too us. This projet allowed us to become familar with all of them. The result is a CI/CD pipeline that combines these various technologies into one process.

The following sections describe how we used each technology in more detail as well as our experience while working on the project.

GitHub

GitHub Link: https://github.com/Bsinglet3/SSW590

Our project starts with GitHub, where we choose to host our code. The benefits of using source control are well known. Our repo provides us with a single location to host our code. It makes it easier to add features and manage sucsessive versions. As mentioned, we were both familar with Git but this project afforded us the opportunity to grow our skills with it. We both made extensive us of Git Bash commands, Ben choose todo so in the terminal while Omar utilized Visual Studio Code's Git integration. Besides it personal benefits, GitHub is also what gives our pipeline a clear trigger: whenever a commit is made to the repo.

DigitalOcean

To host our projet, we choose to use a virtual machine on the DigitalOcean platform. Unlike Azure or AWS which typically charge per amount of computation, DigitalOcean has flat rates. We initially purchased a server with 1 vCPU, 1 GB of RAM, and a 25 GB SSD for $5 a month. In the instance that we overloaded our server, it would simply crash instead of scalling out and charging us more. This was very benefical for a pair of broke college students. Besides flat rates, DigitalOcean also preforms automatic backups and allows us to monitor various server metrics such as CPU and Disk usage. These later features were useful when after getting Jenkins and Docker installed on our server, we did observe some instability. As such, we chose to increase our server RAM to 2 GB, resolving the issues.

Jenkins

The backbone of our project is Jenkins. Hosted on our DigitalOcean server, it is what allows us to automate large portions of the deployment process. Instead of running our pipeline on the built-in node, we created a child node. We encoutered several issues with the built-in node which used it's own jenkins user. The child node logs in as a normal user with sudo permissions, resolving our issues. As mentioned, Jenkins is triggered by commits to our Githuh repo. Specifically, Jenkins is listening for a push Webhook from Github. When triggered, Jenkins will pull the latest version of the repo into the server. Jenkins can then execute shell commands; however, documentation advises that Jenkins calls a shell script instead of running many commands itself. We followed this advice. After the script is complete, Jenkins will check the outcome of the build and if it fails, it will notify us through email.

Bash

During the configuration of our DigitalOcean server, we extensively used Bash commands. The installation of any sofware, such as Jenkins and Docker in the command line as the server is headless. During this installation, we became familiar with the server's file system and how to move through it. Once installation was complete, the use of a Bash script was required to create and deploy the Docker container. A relatively simple bash script is trigger by jenkins that stops and removes and existing docker containers. The script then places the most recent version of the project files into the correct directory. It will then create and start an NGINX container using this directory. Lastly, it checks to make sure that this single, up-to-date container is the only running container and if not, it returns an error, otherwise the script exits normally.

Docker

The website itself is hosted on an NGINX container created by the bash script. The container is set up so that it is persistant and will remain up until manually stopped. The container also exposes port 80 so that the website can be accessed through HTTP. The benefit of hosting NGINX in a container instead of directly on our machine is that it allows us to quickly deploy the latest build of our website. With 3 commands, we can remove the existing container and start the new one. In addition, the utilization of docker containers in this project allowed us to become more familar with how they work and to set them up.
